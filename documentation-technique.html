Documentation Technique de la Ferme Agentique Moulinsart

Vue d'Ensemble

Le système de Moulinsart est une "ferme agentique" conçue pour automatiser l'exécution de projets via une équipe d'agents IA. L'architecture repose sur une communication asynchrone par email, orchestrée par un serveur central qui notifie les agents de nouvelles instructions directement dans leurs terminaux.

L'ensemble est géré et surveillé via un portail web central, le "Portail de Moulinsart".

Architecture du Système

Le système est composé de plusieurs services indépendants mais interconnectés qui fonctionnent en tandem.

Composants Clés

Serveur d'Observabilité Oracle (server/index.ts)

  • Rôle : Cerveau central du système.
  • Fonctions :
    • Fournit une API REST pour gérer les projets, les sprints et les tâches.
    • Maintient l'état du système dans une base de données unique.
    • Diffuse des événements en temps réel à l'interface web via WebSockets.
    • Expose des endpoints pour le workflow de génération de mémoire.

Serveur Mail & Notifications (server/mail-server.ts)

  • Rôle : Système nerveux et de communication.
  • Fonctions :
    • Opère un serveur SMTP (sur le port 1025) qui intercepte tous les emails échangés.
    • Stocke chaque email dans la base de données centrale.
    • Déclenche le "Heartbeat" : notifie l'agent destinataire en injectant un message directement dans sa fenêtre tmux.

Sessions des Agents tmux (moulinsart-manager.sh)

  • Rôle : L'environnement d'exécution des agents.
  • Structure :
    • Deux équipes sont définies : nestor-agents et haddock-agents.
    • Chaque équipe est une session tmux contenant 4 agents, chacun dans son propre panel.
    • Le script moulinsart-manager.sh est l'outil principal pour démarrer, arrêter et gérer ces sessions et les services associés.

Base de Données Centrale (oracle.db)

  • Technologie : SQLite, configuré en mode WAL (Write-Ahead Logging).
  • Rôle : Source de vérité unique. Le mode WAL est critique car il permet au serveur Oracle et au serveur Mail d'écrire dans la base de données simultanément sans conflits de verrouillage.
  • Contenu : Stocke tout : projets, tâches, emails, historique, etc.

Le Workflow de Communication : Le Cœur du Système

Le mécanisme de notification est ce qui permet au système de fonctionner de manière asynchrone et réactive. Voici le déroulement exact :

  1. Émission : Un acteur (le Commandant ou un autre agent) envoie un email à un agent (ex: tintin@moulinsart.local).
  2. Interception : Le serveur SMTP (mail-server.ts) sur le port 1025 reçoit l'email.
  3. Persistance : L'email est immédiatement sauvegardé dans la table emails de la base de données oracle.db.
  4. Notification TMUX (Étape Cruciale) :
    • La fonction injectToTmux est appelée.
    • Elle identifie l'agent destinataire (ex: tintin).
    • Elle détermine la session tmux (nestor-agents) et le panel (1) correspondants grâce à une table de mappage interne.
    • Elle exécute une série de commandes tmux send-keys pour :
      1. Effacer l'écran du terminal de l'agent pour attirer son attention (C-l).
      2. Afficher un message de notification formaté, contenant l'expéditeur et le sujet de l'email.
      3. Fournir des commandes de base prêtes à l'emploi (ex: curl pour lire ses emails).
      4. Attendre 2 secondes, puis envoyer une touche "Entrée" (C-m) pour rafraîchir le prompt de l'agent et le rendre à nouveau actif.
  5. Activation de l'Agent : L'agent, qui est en attente dans sa ligne de commande, reçoit cette notification comme s'il s'agissait d'une entrée utilisateur. Il peut alors exécuter les commandes suggérées pour prendre connaissance de sa nouvelle tâche.

Ce cycle constitue le "battement de cœur" (heartbeat) du système, transformant une communication passive (email) en une action directe et ciblée.

Le Workflow de Mémoire Post-Sprint

Après la fin d'un sprint, le système dispose d'un workflow pour synthétiser les événements et mettre à jour la "mémoire" des agents. Ce processus est initié depuis l'onglet "Memory" du portail web Oracle.

Point Clé : La "génération" de mémoire est un processus déterministe. Il n'utilise pas de modèle de langage externe (IA). Il s'agit d'une synthèse structurée et factuelle des données collectées durant le sprint depuis la base de données, garantissant rapidité, fiabilité et prévisibilité.

4.0. Interface Memory - Fonctionnalités

Onglet Memory du Portail Oracle (http://localhost:5175)

  • Filtrage par projet : Sélection du projet et du sprint concerné
  • Timeline fusée animée : Visualisation en temps réel du workflow 4-étapes
  • Bouton "Generate Memory" : Déclenche le pipeline complet automatique
  • Fichiers générés affichés : PROJECT_MEMORY.md et CLAUDE.md modifiés
  • Statistiques de purge : Nombre d'emails marqués comme lus

4.2. Étapes Détaillées

  1. Génération de la Synthèse (/api/memory/generate)
    • Source de données : Uniquement les tables projects et tasks de la base de données.
    • Processus : Le serveur récupère toutes les tâches du sprint, les classe, et génère un résumé structuré (objectifs, tâches finies, problèmes) sous forme d'objet JSON.
  2. Écriture de la Mémoire Projet (/api/memory/write-project)
    • Fichier cible : /projects/NOM_PROJET/PROJECT_MEMORY.md. Un seul fichier par projet.
    • Logique anti-doublons : Avant d'écrire, le système vérifie si un en-tête pour le sprint (ex: ## Sprint 8) existe déjà dans le fichier. Si oui, l'écriture est annulée pour éviter les duplications.
  3. Mise à Jour des Agents (/api/memory/write-agents)
    • Fichiers cibles : Le fichier CLAUDE.md de chaque agent ayant participé au sprint.
    • Logique anti-doublons : Le système vérifie également si la mémoire de ce sprint a déjà été ajoutée au CLAUDE.md de l'agent. Si oui, l'agent est ignoré.
    • Action : Un court résumé des tâches de l'agent pour ce sprint est ajouté à la fin de son fichier CLAUDE.md.
  4. Purge des Données (/api/memory/purge-mails)
    • Action : Le système détermine les dates de début et de fin du sprint en se basant sur les tâches.
    • Il marque ensuite tous les emails (en passant le champ deleted à 1) dans la base de données dont la date se situe dans cet intervalle. C'est un "soft delete", les données ne sont pas réellement effacées mais sont cachées.

Améliorations Techniques Récentes

Cette section documente les corrections critiques apportées au système pour assurer sa stabilité et fiabilité.

5.1. CORS Policy - Sécurité Zero

Problème résolu : Blocage des requêtes DELETE

  • Symptôme : Erreur "Method DELETE is not allowed by Access-Control-Allow-Methods" lors de la suppression de projets
  • Cause : Headers CORS restrictifs dans server/index.ts
  • Solution : Configuration CORS zero-sécurité :
const headers = {
  "Access-Control-Allow-Origin": "*",
  "Access-Control-Allow-Methods": "*",
  "Access-Control-Allow-Headers": "*",
  "Access-Control-Allow-Credentials": "true",
  "Access-Control-Max-Age": "86400"
};

5.2. Script Manager Brutal

moulinsart-manager.sh - Version Renforcée

  • Fonction brutale : restart_services() modifiée
  • Actions :
    1. MASSACRE TOTAL : pkill -9 tous processus Bun/Node/Vite
    2. LIBÉRATION PORTS : lsof + kill -9 sur 3001, 1025, 1080, 5175
    3. REDÉMARRAGE PROPRE : Oracle (3001), Mail (1025/1080), Web (5175)
    4. VÉRIFICATION : État final des ports
  • Résultat : Plus jamais de conflits "port already in use"

5.3. Configuration SQLite WAL Critique

Configuration obligatoire dans server/index.ts ET server/mail-server.ts : 
db.run("PRAGMA journal_mode = WAL");
db.run("PRAGMA synchronous = NORMAL");
db.run("PRAGMA cache_size = 1000");
db.run("PRAGMA temp_store = memory");
Sans cette configuration, conflits de verrouillage garantis entre Oracle et Mail servers.

5.4. Diagnostic et Maintenance

Commandes de Vérification

  • État des services : curl -s http://localhost:3001/health
  • Ports actifs : lsof -i :3001,1025,1080,5175
  • Mode SQLite : sqlite3 data/oracle.db "PRAGMA journal_mode"
  • Logs temps réel : tail -f /tmp/oracle.log
